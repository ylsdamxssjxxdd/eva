# 11 系统架构师：代理的代理

[返回总览](GitHubPages用户引导.md) · [上一页：使用技能](10-使用技能.md) · [下一页：机体控制](12-机体控制.md)

## 🎯 适用场景
- 需要架构设计、方案推演、接口定义与分工计划。
- 让模型先给“设计稿”，再交由系统工程师落地。

## 📦 你会得到
- 切换到架构师角色的提问模板。
- 结构化输出（目标/架构/接口/风险/计划）的示范。

## 🛠️ 前置准备
1. 在“约定”选择“系统架构师”角色，或在提示词中强调“先设计后执行”。
2. 准备需求、约束（性能/安全/成本）与验收标准。

## 🚀 操作步骤
1. 在对话描述背景、目标、约束、现有资产（接口/数据库/服务）。
2. 要求模型输出：
   - 目标与范围
   - 架构/流程（文字描述即可）
   - 模块划分与接口草案
   - 风险与备选方案
   - 执行计划（交给系统工程师）
3. 如需代码样例或 API 契约，后续继续追问细化。

## 🧠 小技巧
- 约束越清晰，设计越贴合：写明 QPS、延迟、数据量、安全要求。
- 想对比多方案：要求“列 2-3 个可选方案，再推荐其一并说明取舍”。
- 希望沉淀：让模型按文档格式输出，直接放入 README/设计稿。

## ❓ 常见问题
- 输出太泛：在提问中指定“结合现有技术栈/限制”，或要求表格化。
- 方案不可落地：让模型基于“当前资源清单”重新评估。
- 文字太长：要求分段或列点精简。
