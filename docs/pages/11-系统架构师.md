# 11 系统架构师：规划与设计模式

[返回总览](../GitHubPages用户引导.md) · [上一页：使用技能](10-使用技能.md) · [下一页：机体控制](12-机体控制.md)

## 适用场景
- 需要架构设计、方案推演、接口定义与分工计划。
- 让模型先给出“设计稿”，再交由系统工程师落地。

## 你会得到
- 切换到架构师角色的提示词思路。
- 让输出更可执行的提问模板。

## 前置准备
1. 在“约定”选择“系统架构师”角色（若有），或在系统提示词中强调“先设计后执行”。
2. 准备需求、约束（性能/安全/成本）、目标与验收标准。

## 操作步骤
1. 在对话中描述背景、目标、约束与已有资产（现有接口/数据库/服务）。
2. 要求模型输出以下结构：
   - 目标与范围
   - 架构或流程图（文字描述即可）
   - 模块划分与接口草案
   - 风险与备选方案
   - 后续执行计划（交给系统工程师）
3. 如需代码样例或 API 契约，可在后续追问细化。
4. 确认方案后，转到“系统工程师”页面的流程去实施。

## 进阶提示
- 约束越清晰，设计越贴合：明确 QPS、延迟、数据量、安全要求。
- 想对比多方案：要求“先列 2-3 个可选方案，再推荐一个并说明取舍”。
- 需要沉淀：让模型输出可直接放入 README/设计文档的格式。

## 常见问题
- 输出太泛：在提问里指定“结合现有技术栈/限制”，或要求表格化输出。
- 方案不可落地：让模型基于“当前资源清单”重新评估。
- 文字太长：要求分段返回或列点精简。

## 验收要点
- 获得结构化、可实施的设计稿，包含接口/模块/风险。
- 方案能无缝交接给工程师执行。
